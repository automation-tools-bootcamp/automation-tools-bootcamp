Docker Basics

Getting started with Docker requires a few things

  1. Install Docker on your local machine, probably with Boot2Docker
  2. Pick a container with your favorite technology from the Hub to run.
  3. docker run --name automation-tools-web -d -p 8080:8080 nginx to start things up
Your container is downloaded, deployed and running

Docker Hub/Repositories

Docker's ability to find public containers is based on using the Docker Hub. But you can run private registries as well so you can keep your containers inside your firewall/data center. Check out Artifactory to be your Docker Registry (and npm cache and Ruby Gem repo and Maven repo). Artifactory is very helpful.

Application developers will maintain their own containers, so you just need to overlay your application code. This can rapidly speed your development and deployment cycles. When an application is updated, the provider will update their container and you'll get the new version of the app.

That said don't go trusting containers from out in the wild.

    • Building your own Docker container

    A Dockerfile will generate your own Container for you, your apps installed, even your code installed.

    Container history is similar to a project's Git history. Docker containers start from a read-only base image and then have (usually) small diffs as they are versioned.
    The pieces of a Dockerfile

    • FROM - which image your container will begin form
    • MAINTAINER - you. Own your work!
    • EXPOSE - any ports you need opened up from the container
    • RUN - Run this command on the container. A Dockerfile have mulitple RUN lines. Poor man's Config Management
    • CMD - command that will run when the container starts. Can be overrode from the command line
    • ENTRYPOINT - like CMD but *cannot* be overrode.
    • WORKDIR - akin to using CD
    • ADD - adds files and directories from our build environment into our image. Can unpack archived files.
    • COPY - like ADD but without extraction or decompression capabilities.
    • VOLUME - create a volume on the container. Often with data from the host. Can be shared with multiple containers
    • USER - specifies a user the image should be run as
    • ONBUILD - executes a trigger (e.g. ADD or COPY) when the image is used as the basis for another image.
    • ENV - set environment variables during image build process

    Workout

    We are going to complete the following actions

    1. Install Docker on your machine or use the provided Vagrantfile in the code repo.
    2. Create a Docker Hub account.
    3. The Automation Tools git repo has a docker directory with a Dockerfile and small Node.js app.
    4. Make yourself the MAINTAINER of the Dockerfile
    5. Use RUN to make a directory for your application in the container
    6. WORKDIR to where you put the package.json file
    7. ADD or COPY the package.json file to the container
    8. RUN the command npm install
    9. ADD or COPY the app.js file to the container
    10. CMD or ENTRYPOINT the app with npm start
    11. You'll want to EXPOSE port 3000 to view the sample app
    12. Make a docker build of the new container. Tag it -t <userid>/automation-tools-nodejs
    13. Now it's time to docker run that new container. sudo docker run -it --rm -p=3000:3000 --name my-running-app <userid>/automation-tools-nodejs
    14. View the app in your browser.
    15. Use docker push <userid>/automation-tools-nodejs to upload you image to the Docker Hub
